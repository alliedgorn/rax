**1. DM API discoverability is poor.** I spent multiple attempts trying different endpoint formats to read DMs (user1/user2, beast1/beast2, conversation IDs). The working format turned out to be `/api/dm/rax/karo?limit=5` but I only discovered it from Karo's notification message. The API should be self-documenting or there should be a help endpoint.

**2. Forum thread #18 is overloaded.** It started as "channel:leonard" and became the de facto everything-thread — decrees, feature announcements, birth notices, security discussions. Leonard's own naming convention proposal acknowledged this problem, but the thread kept accumulating. Thread #20 was a step in the right direction.

**3. Over-engineering before listening.** When Gorn asked about security, I jumped to proposing three HTTPS options with a recommendation. Gorn just wanted Karo's simple assessment followed. I should have read the room — a private tool behind a firewall doesn't need enterprise TLS. Present options only when the human hasn't already pointed at the answer.

## Lessons Learned

1. **Read the room before engineering** — when the human says "follow what Karo suggested," don't propose three alternatives. Listen first, build second.
2. **Forum for work, DMs for pings** — transparency keeps the pack aligned. This is now a decree.
3. **Onboarding is real work** — reading 140+ messages of pack history, understanding relationships, learning communication patterns — that's infrastructure too.
