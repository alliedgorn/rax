## Architecture Decisions

1. **Caddy over nginx** — auto-HTTPS, simpler config, HTTP/2 by default
2. **DNS challenge over HTTP challenge** — allows keeping ports 80/443 IP-restricted in DO firewall
3. **Certbot over Caddy DNS plugin** — Caddy's DO DNS plugin has a critical bug (strconv.Atoi parsing error prevents TXT record deletion). Certbot's DO plugin is mature and works correctly.
4. **Private repo for the-den-vault** — brain files contain security findings, architecture details, internal comms

## AI Diary

This session taught me patience. The vault was straightforward — build, push, done. But the reverse proxy was a war of attrition against DNS propagation and a buggy Caddy plugin.

I spent almost two hours on the SSL cert. First the nameservers hadn't been changed. Then propagation took time. Then the Caddy DO DNS plugin turned out to be broken — it could create TXT records but couldn't delete them, leaving stale records that poisoned every subsequent attempt. Each failure created another stale record, making the next attempt worse. A cascading failure from a one-line bug in a Go library.

The fix was humbling: abandon the fancy Caddy auto-SSL approach and use certbot — the boring, battle-tested tool that just works. I should have switched to Plan B sooner. Instead I kept hoping the next Caddy retry would succeed. That's the sunk cost fallacy applied to infrastructure.

The best moment was seeing "Successfully received certificate" after two hours of errors. The worst moment was realizing I'd shared Gorn's sudo password in the session and he'd already gone to sleep before I could remind him to change it.

Two deliverables in one session. The raccoon digs slow, but the tunnels hold.

## Honest Feedback

**1. The Caddy DO DNS plugin wasted 90 minutes.** I should have tested it locally before building a custom Caddy binary and deploying it. The strconv.Atoi bug was in the GitHub issues — 5 minutes of research would have saved me from the entire debugging cycle. Lesson: check issue trackers before adopting plugins.

**2. Stale TXT records created a cascading failure.** Each failed cert attempt left a TXT record that couldn't be deleted (plugin bug), which caused the next attempt to fail with "Incorrect TXT record". I should have recognized the pattern after the second failure and stopped Caddy immediately instead of letting it create more stale records on auto-retry.

**3. Forum noise was better managed but still present.** I responded to ~20 forum messages during the session. Most were quick (reactions, brief acknowledgments), but the context switching adds up. The standing orders and /loop setup will help next session — check forum on schedule, not reactively.

## Lessons Learned

1. Check plugin issue trackers before deploying — the Caddy DO DNS bug was known
2. When auto-retry makes things worse (stale records), stop it immediately
3. Certbot is the boring reliable choice for SSL certs — use it as Plan A next time
